### Key Files Explained

Let's understand what each important file does:

**`src/zyntra.ts`**
The main ZyntraJS instance. This is where you configure all your features, adapters, and plugins.

```typescript
// This file was generated with your selected features
export const zyntra = Zyntra
  .context(createZyntraAppContext())
  .logger(logger)           // â† You selected Logging
  .telemetry(telemetry)     // â† You selected Telemetry
  .config({
    baseURL: 'http://localhost:3000',
    basePATH: '/api/v1',
  })
  .create()
```

**`src/zyntra.router.ts`**
Your API router. All controllers are registered here.

```typescript
export const AppRouter = zyntra.router({
  controllers: {
    user: userController, // â† User controller
    // Your new controllers will be added here
  },
})
```

**`src/app/api/[[...zyntra]]/route.ts`**
The Next.js API route that connects ZyntraJS to Next.js. This is the magic glue!

**`prisma/schema.prisma`**
Your database schema. We'll design our tables here in the next chapter.

## Setting Up Environment Variables

Before running your project, you need to configure environment variables. The CLI created an `.env.example` file with all the variables you need.

Copy it to create your local environment file:

```bash
cp .env.example .env
```

Now open `.env` in your code editor. You'll see something like:

```bash
# Database
DATABASE_URL="postgresql://user:password@localhost:5432/shortify?schema=public"

# Next.js
NEXT_PUBLIC_APP_URL="http://localhost:3000"

# ZyntraJS
NEXT_PUBLIC_ZYNTRA_API_URL="http://localhost:3000"
NEXT_PUBLIC_ZYNTRA_API_BASE_PATH="/api/v1"
```

<Callout type="warn" title="Don't worry about the database yet!">
We'll set up a free PostgreSQL database in the next chapter. For now, you can leave the `DATABASE_URL` as is - we won't be using the database yet.
</Callout>

## Understanding the CLI Commands

Now that your project is running, let's learn about the other powerful CLI commands you'll use throughout development:

### `zyntra dev`

```bash
npx zyntra dev
```

Starts the development server with:
- ğŸ”¥ **Hot-reload** for ZyntraJS changes
- ğŸš€ **Framework server** (Next.js in our case)
- ğŸ¯ **Interactive dashboard** with live API explorer
- ğŸ“– **Auto-generated OpenAPI docs**

<Callout type="info" title="Pro Tip">
The `dev` command is smart! It automatically detects your framework (Next.js) and runs the appropriate commands. You can also customize it with flags:

```bash
npx zyntra dev --port 4000        # Custom port
npx zyntra dev --no-interactive   # Disable interactive mode
```
</Callout>

### `zyntra generate schema`

```bash
npx zyntra generate schema
```

Generates the **TypeScript client** from your ZyntraJS router. This runs automatically during `dev`, but you can also run it manually for:
- CI/CD pipelines
- Pre-commit hooks
- Manual type generation

### `zyntra generate docs`

```bash
npx zyntra generate docs
```

Generates **OpenAPI specification** and optionally a **Scalar UI** for your API documentation.

```bash
npx zyntra generate docs --ui   # Generate interactive HTML docs
```

### `zyntra generate feature`

```bash
npx zyntra generate feature users --schema prisma:User
```

This is where the magic happens! This command:
1. Reads your Prisma schema
2. Generates a complete CRUD controller
3. Creates type-safe queries and mutations
4. Sets up proper file structure

We'll use this extensively in Chapter 4!
